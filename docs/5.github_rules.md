## 開発ルールとチームの心構え 🤝

短期集中開発を成功させるため、以下のルールを定めます。

---

### 1. Git / GitHub の運用ルール

チーム開発における交通ルールです。

### 🎫 Issue 駆動開発の導入

これからの開発は、**「Issue（=タスクのチケット）」を起点**に進めます。これにより、「誰が」「何を」「いつまでに」やっているのかが明確になります。

- **Issue の作り方:**

  - 作業を始める前に、必ず GitHub で Issue を作成します。
  - **Title:** 「〇〇を作成する」「△△ を修正する」など、タスク内容が具体的にわかる名前をつけます。
  - **Description:** 「何を」「なぜ」やるのか、どうなれば「完了」なのか（完了条件）を簡潔に書きます。
  - **Assignees:** 自分、または担当者を設定します。

- **ブランチ戦略 (Branching Strategy)**

  - `main` **ブランチ:** 常に正常に動作する**神聖なブランチ**です。直接の`push`は禁止します。
  - **作業ブランチ:** `main`から新しいブランチを作成して作業します。
    - 命名規則: `feature/担当者名/機能名` (例: `feature/tanaka/create-header`)

- **開発の流れ (Workflow)**

  1.  作業開始前に、`git pull origin main`で`main`ブランチを最新の状態にします。
  2.  自分の作業ブランチで開発を進め、こまめにコミットします。
  3.  作業が完了したら GitHub に`push`します。
  4.  **プルリクエスト (Pull Request)**を作成し、パートナーにレビューを依頼します。
  5.  レビューで承認されたら、`main`ブランチにマージします。

- **コミットメッセージのルール (Commit Message Convention)**
  コミット履歴を分かりやすくするため、メッセージに接頭辞をつけます。
  **フォーマット:** `種類: 何をしたか`

| 種類 (Prefix) | 説明                                         |
| :------------ | :------------------------------------------- |
| **feat**      | **新機能**の追加                             |
| **fix**       | **バグ**の修正                               |
| **style**     | コードの**見た目**の変更（フォーマットなど） |
| **refactor**  | 機能を変えないコードの**改善**               |
| **docs**      | **ドキュメント**の修正（README など）        |
| **chore**     | ビルドやツールなど**雑多な**タスク           |

---

### 2. コーディングスタイル

コードの書き方を統一し、可読性を高めます。

- **フォーマッター:** **Prettier**を導入し、ファイル保存時に自動でフォーマットがかかるように設定します (`formatOnSave`)。
- **リンター:** **ESLint**（Next.js 標準）を利用し、エラーや警告が表示された場合はそれに従って修正します。

✅ **合意事項:** 「Prettier と ESLint の指摘には素直に従う」

---

### 3. 各担当者が意識すべきこと 💡

API 仕様書は、フロントエンドとバックエンドをつなぐ「**契約書**」です。

#### フロントエンド担当 (A さん) の心構え

- **モックデータ活用:** バックエンドの API が完成する前に、偽のデータを使って UI の実装を先行させます。
- **ユーザー視点:** ユーザーの操作（正常系、異常系）を想像しながら実装を進めます。
- **契約遵守:** API 仕様書に定められた通りのリクエストデータを送信します。

#### バックエンド担当 (B さん) の心構え

- **Postman での事前テスト:** フロントエンドと結合する前に、Postman で API が仕様書通りに動作することを徹底的に確認します。
- **丁寧なエラー処理:** エラー発生時に、フロントエンド担当者が原因を理解しやすいエラーメッセージを返します。
- **契約遵守:** API 仕様書に定められた通りのレスポンスデータを返却します。

#### 二人で常に意識すること

API 仕様書に変更が必要になった場合は、**必ず二人で話し合って合意してから**作業を進めます。
